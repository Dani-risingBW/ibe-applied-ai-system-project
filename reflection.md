# Library Room Booking System Reflection

## 1. How AI Was Used During Development

I used AI throughout implementation for:
- Architecture planning (component boundaries for UI, booking engine, adapters, and AI module)
- Structured prompting design for natural-language booking parse
- Debugging integration issues between Streamlit state, booking flow, and persistence
- Test generation and iterative refinement for guardrails and edge cases

Most helpful prompt style:
- Prompting for a phase plan to break up task then implementing them one by one using specific prompting
- Asking for failure modes and test cases before coding changes

## 2. Helpful vs Flawed AI Suggestions

### Helpful AI Suggestion
A strong suggestion was to enforce guardrails in the booking engine (ConflictDetector) rather than only in the UI.

Why it helped:
- It made booking rules reusable and consistent.
- It prevented invalid bookings even if UI flow changes later.
- It improved testability for constraints like overlap, duration limits, and past-time blocking.

### Flawed AI Suggestion
A flawed suggestion was to rely on a single short AI suggestion response without handling truncation.

Why it was flawed:
- Responses occasionally appeared cut off in UI (for example, "AI Suggestion: Since your...").
- It degraded user clarity.

How it was corrected:
- Increased token budget for suggestion generation.
- Added robust response extraction.
- Added retry logic when output looks incomplete.

## 3. System Design Reflection

The most important design decision was keeping clear module boundaries:
- app.py handles interaction and state
- booking_engine.py handles booking rules and lifecycle
- adapters/ handles availability sources
- ai_assistant.py handles AI parse/suggestion/coaching behavior
- persistence.py handles durable booking storage

This separation made it easier to add AI extensions without breaking core booking logic.

## 4. Reliability and Testing Reflection

Reliability improved through practical guardrails:
- Past-date/time booking block
- Room overlap checks
- Library policy enforcement (hours, limits)
- AI prompt coaching for non-booking input

Testing strategy:
- Unit tests for booking rules and AI helpers
- Focused edge-case tests for conflict and guardrail behavior
- Regression checks after major AI integration changes

## 5. Current Limitations

- Availability quality depends on adapter configuration and source quality.
- AI output can still vary in phrasing despite deterministic prompts.
- Scraper-based integrations may require site-specific selector maintenance.
- No full automated quality benchmark report is generated by default for all user flows.
